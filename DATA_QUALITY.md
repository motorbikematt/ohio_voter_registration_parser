# DATA_QUALITY.md — Ohio heterogeneity registry (per-county data-source deviations)

**This is the single registry for the "88-county assumption" (CLAUDE.md §5).** The SWVF and every stream after it are 88 sovereign Boards of Elections under one fixed schema that normalizes columns but not meaning. Every *known* per-county deviation — a place a county encodes the same fact differently than its neighbors — is recorded here so a resolver can cite its row instead of re-discovering it as a bug. When a new mole appears, **read this file first**: a deviation already listed gets a resolver that cites it; a new one gets a bounded per-county profile (`tools/admin/data_profile.py::profile_column`, §7) and a new row. Never open a second registry — drift between two ledgers is the disease §5 forbids.

**Divergence spans four levels** (the `Defect class` column names which): **value** (same column, different meaning — `WARD="CANTON"` = township, not a ward), **schema** (column present here, absent there), **format** (the concept arrives as CSV here, PDF/HTML/zip there), **provenance** (which artifact is authoritative is itself unknown). **Stream** = which dataset the deviation lives in; the auto-block below is entirely the `jurisdiction` stream (SWVF jurisdiction columns). New streams (elected-officials, results, committees) append their own rows in the manual section — or their own auto-block once a validator measures them — under the same schema.

The `jurisdiction`-stream table below is regenerated by `python tools/admin/validate_jurisdiction_fields.py --ledger` (do not hand-edit inside the `auto:` markers); edit only the manual section beneath it. Note: the validator re-derives ward/place assignments *live* from the current resolver code, so this block goes stale whenever a resolver changes even if the parquet does not — regenerate it after resolver edits.
